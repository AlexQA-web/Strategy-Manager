# Известные проблемы и решения

Документ для Claude Code: при встрече с описанными симптомами — применять готовое решение, не искать новое.

---

## 1. Race condition: двойной вход в позицию

**Симптом:** LiveEngine открывает позицию дважды при быстром появлении нескольких баров подряд.

**Причина:** `_order_in_flight` и `_position` проверялись в разных критических секциях.

**Решение:** Оба флага проверяются и устанавливаются **внутри одного** `with self._position_lock`. Никогда не разбивать эту проверку на две отдельные блокировки.

```python
with self._position_lock:
    if self._position != 0:
        return  # уже в позиции
    if self._order_mode == "limit":
        if self._order_in_flight:
            return  # уже есть ордер
        self._order_in_flight = True  # устанавливаем здесь же
```

## 2. Блокировка GUI при get_history() / get_sec_info()

**Симптом:** Интерфейс зависает на несколько секунд при открытии графика или вкладки лотности.

**Причина:** QUIK-коннектор делает синхронный сетевой запрос в GUI-потоке.

**Решение:**
- `chart_window.py` → `DataLoader(QThread)` с `_cancelled` флагом и таймаутом 120с
- `main_window.py` → `_get_sec_info_cached()` запускает фоновый поток через `_refresh_sec_info_background()`, возвращает кэш немедленно
- **Никогда** не вызывать `connector.get_history()` или `connector.get_sec_info()` напрямую из GUI-потока

---

## 3. Потеря fills при отмене chase-ордера

**Симптом:** После `cancel_order()` часть исполненного объёма не учитывается.

**Причина:** watcher отписывался до прихода финального статуса с fills.

**Решение:** В `chase_order.py` метод `_cancel_and_wait()` — сначала ждём финального статуса (`_wait_for_terminal_status`, таймаут 2с), **потом** вызываем `unwatch_order()`. Порядок нельзя менять.

## 4. NaN в индикаторах на первых барах

**Симптом:** `on_bar` получает `None` или `nan` в `current.get("_sma")` и падает.

**Причина:** `rolling(min_periods=period)` возвращает NaN пока не накоплено достаточно баров.

**Решение:** Всегда проверять перед использованием:
```python
val = current.get("_indicator")
if val is None or val != val:  # None или NaN (NaN != NaN)
    return {"action": None}
```

## 5. Некорректный point_cost от TransAQ DLL

**Симптом:** PnL рассчитывается неправильно для фьючерсов (в разы больше/меньше реального).

**Причина:** TransAQ DLL возвращает некорректный `point_cost` для ряда инструментов (особенно валютные фьючерсы).

**Решение:** В `finam_connector.get_sec_info()` — приоритет MOEX API над данными DLL. Порядок:
1. Запрос к `moex_client.get_instrument_info(ticker, "futures")`
2. Если вернул `point_cost` — использовать его
3. Fallback — данные DLL

**Не менять** этот приоритет без явной причины.

## 6. Дублирование сделок в order_history

**Симптом:** Одна сделка записывается несколько раз, PnL задваивается.

**Причина:** DLL присылает callback `OnTrade` несколько раз для одного `tradeno`.

**Решение:** В `finam_connector._parse_trades()` — дедупликация через `_processed_trades: dict[tradeno, timestamp]` под `_processed_trades_lock`. Очистка старше 24ч каждый час.

## 7. QScrollArea «съедает» wheel events у SpinBox

**Симптом:** QSpinBox или QComboBox внутри QScrollArea меняет значение при прокрутке страницы.

**Решение:** Для всех числовых виджетов в скролл-областях использовать кастомные классы:
```python
class _NoScrollSpinBox(QSpinBox):
    def wheelEvent(self, event):
        if not self.hasFocus():
            event.ignore()
            return
        super().wheelEvent(event)
```
Аналогично для `QDoubleSpinBox` и `QComboBox`. Примеры в `commission_settings_widget.py` и `commission_preview_dialog.py`.

## 8. Deadlock при остановке LiveEngine

**Симптом:** `engine.stop()` зависает навсегда.

**Причина:** `_chase_lock` захватывался внутри `_position_lock`, а в другом потоке — наоборот.

**Решение:** В `stop()` — копируем список chase-ордеров **вне** `_chase_lock`, затем отменяем. В `_on_chase_done()` — сначала освобождаем `_chase_lock`, затем захватываем `_position_lock`. Никогда не вкладывать эти два lock'а.

---

## 9. get_history() возвращает None при первом запуске

**Симптом:** График пустой сразу после подключения, хотя коннектор подключён.

**Причина:** DLL ещё не получила список инструментов от сервера (асинхронная инициализация).

**Решение:** В `chart_window.py` — `chart_cache` возвращает кэш с диска немедленно, пока коннектор прогружается. Если кэша нет — показывать заглушку (текст "Нет данных"), не ошибку.

---

## 10. strategies.json: потеря данных при конкурентной записи

**Симптом:** После одновременного сохранения двух стратегий — данные одной пропадают.

**Причина:** `save_strategy()` делает read-modify-write без lock'а.

**Решение:** В `storage.py` все write-операции под `_write_lock`. Для read-modify-write одной записи использовать `save_setting()` (захватывает lock и читает с диска, минуя кэш через `use_cache=False`).

---

## 11. Таймаут get_history() в QUIK при большом lookback

**Симптом:** LiveEngine логирует "get_history завис (>30 сек), пропускаем тик".

**Причина:** QUIK DataSource с `count=0` (все бары) медленно отвечает для длинных историй.

**Решение:** В `live_engine._load_and_update()` — фоновый поток с `join(timeout=30)`. Если завис — пропускаем тик, не блокируем poll_loop. Уменьшить `days` в `get_lookback()` стратегии.

---

## 12. Мерцание таблицы агентов при обновлении

**Симптом:** Главная таблица мерцает каждые 5 секунд.

**Причина:** `_refresh_table()` пересоздаёт все строки полностью.

**Текущий статус:** Известная проблема, не исправлена. При изменении — использовать `setItem()` для обновления только изменившихся ячеек вместо полного пересоздания. Не трогать структуру `_refresh_table()` без необходимости — сложная логика с дочерними строками корзины инструментов.

---

## 13. `on_precalc` с merge создаёт дублированные колонки

**Симптом:** После `df.merge(...)` появляются колонки `_sma_x` и `_sma_y` вместо `_sma`.

**Причина:** Повторный вызов `on_precalc` на df, который уже содержит эти колонки.

**Решение:** Перед merge удалять старые колонки или использовать `df.loc[df_strat.index, col] = ...` для инкрементального обновления (как в `chart_window._apply_precalc()`).

---

## 14. Финам DLL: мистическое зависание Initialize()

**Симптом:** `finam_connector.connect()` зависает намертво при повторном вызове.

**Причина:** DLL уже инициализирована, второй вызов `Initialize()` блокируется.

**Решение:** Флаг `self._initialized` — проверять перед вызовом. `Initialize()` вызывается **один раз** за весь жизненный цикл приложения. При переподключении использовать только `SendCommand("connect")`.
