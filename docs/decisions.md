# Архитектурные решения (ADR)

Документ объясняет **почему** приняты ключевые решения.
Читать перед тем как предлагать «улучшения» архитектуры.

---

## 1. Почему TransAQ через ctypes, а не COM или REST

**Контекст:** Финам предоставляет коннектор txmlconnector64.dll.

**Решение:** ctypes (`ctypes.WinDLL`), callback через `WINFUNCTYPE`.

**Причины:**
- COM требует регистрации в реестре и pythoncom — хрупко, зависит от окружения
- REST API у Финам есть, но имеет другие ограничения по частоте и функционалу
- DLL даёт полный функционал включая push-уведомления через callback (ордера, позиции, котировки)
- ctypes не требует дополнительных зависимостей, работает в PyInstaller-сборке

**Не менять** на COM или REST без явного требования пользователя.

---

## 2. Почему JSON, а не SQLite или PostgreSQL

**Контекст:** Хранение настроек, стратегий, истории ордеров.

**Решение:** JSON-файлы в `data/`.

**Причины:**
- Десктопное приложение без сервера — SQLite избыточен для объёмов данных
- JSON читается человеком, редактируется вручную при необходимости
- Атомарная запись через `.tmp` → rename решает проблему коррупции
- Резервная копия в `.bak` при каждой записи
- Нет схемы → нет миграций при добавлении полей
- `order_history.json` растёт медленно (сотни записей в год, не миллионы)

**Не предлагать** переход на SQLite/PostgreSQL без явного обоснования объёмами.

---

## 3. Почему polling в LiveEngine, а не подписка на события

**Контекст:** LiveEngine получает новые бары и принимает торговые решения.

**Решение:** Периодический опрос `get_history()` (polling) вместо подписки на свечи.

**Причины:**
- Финам DLL: подписка на свечи (`subscribe_candles`) присылает только текущую — без гарантии получения всех промежуточных баров при реконнекте
- QUIK: `get_candles_from_data_source` возвращает полную историю — удобнее для lookback
- Polling восстанавливается после реконнекта автоматически (просто опрашивает снова)
- Polling позволяет единообразно работать с обоими коннекторами
- Интервал опроса (15–120 сек) достаточен для таймфреймов от 1 минуты

**Не переписывать** на push/подписку без решения проблемы восстановления после реконнекта.

## 4. Почему point_cost берётся с MOEX API, а не из DLL

**Контекст:** Расчёт PnL для фьючерсов: `(price - entry) × qty × point_cost`.

**Решение:** Приоритет MOEX ISS API → fallback на данные DLL.

**Причины:**
- TransAQ DLL часто возвращает некорректный `point_cost` для валютных фьючерсов (Si, CR)
- MOEX API возвращает `STEPPRICE` (стоимость минимального шага) — делим на `MINSTEP` → точный `point_cost`
- Кэш в `MOEXClient` (4ч) минимизирует сетевые запросы
- Проблема воспроизводилась многократно на реальных данных

**Не менять порядок** приоритетов в `finam_connector.get_sec_info()` и `live_engine._load_point_cost()`.

## 5. Почему регистрация коннекторов отложена (не при импорте)

**Контекст:** `finam_connector` и `quik_connector` создаются как синглтоны в конце модулей.

**Решение:** `register_connectors()` вызывается явно из `autostart.py` после инициализации UI.

**Причины:**
- При импорте модуля `FinamConnector.__init__()` не делает сетевых вызовов — это безопасно
- Но регистрация в `connector_manager` + `configure_all()` читает `settings.json`
- `settings.json` должен существовать к этому моменту (создаётся при инициализации `DATA_DIR`)
- Тесты могут импортировать `core.*` без реального окружения — отложенная регистрация позволяет это

## 6. Почему ChaseOrder вместо простой лимитки

**Контекст:** Исполнение ордеров по типу `"limit"`.

**Решение:** `ChaseOrder` — фоновый поток, следит за стаканом и переставляет лимитку.

**Причины:**
- Фьючерсы Si/RI имеют bid-ask spread до 10–20 пунктов в моменты волатильности
- Простая лимитка по last price может не исполниться до закрытия сессии
- ChaseOrder ставит лимитку по bid (для покупки) → при изменении bid переставляет
- Исполнение через стакан экономит на spread по сравнению с рыночной заявкой

## 7. Почему _InstrumentsWidget открывает отдельный диалог

**Контекст:** Параметр типа `"instruments"` (корзина инструментов в Achilles).

**Решение:** Кнопка «Редактировать» → `InstrumentsEditor(QDialog)`.

**Причины:**
- Корзина может содержать 5–20 инструментов — встроенная таблица растянула бы вкладку Параметры
- Каждая строка содержит `TickerSelector` (тяжёлый виджет с автозагрузкой) — lazy init в диалоге
- Диалог позволяет добавлять/удалять строки без перестройки всей формы параметров

## 8. Почему Catppuccin Mocha, а не стандартная Qt тема

**Контекст:** Визуальное оформление приложения.

**Решение:** Кастомный QSS на основе Catppuccin Mocha palette.

**Причины:**
- Стандартная Qt тема выглядит устаревшей на Windows 11
- Catppuccin — популярная dark-тема с хорошим контрастом для финансовых данных
- Зелёный (#a6e3a1) = прибыль/лонг/подключён; красный (#f38ba8) = убыток/шорт — интуитивно
- QSS определён в нескольких местах — менять только через константы в соответствующих файлах

**Не заменять** цвета без согласования всей цветовой системы.

---

## 9. Почему _SettingsMixin вместо наследования от QDialog

**Контекст:** Настройки нужны и как отдельный диалог, и как вкладка в главном окне.

**Решение:** `_SettingsMixin` содержит логику UI, два класса наследуют его:
- `SettingsWindow(QDialog, _SettingsMixin)` — для `exec()`
- `SettingsWidget(QWidget, _SettingsMixin)` — для встраивания в `QTabWidget`

**Причины:**
- До Claude Code 1.x настройки были только диалогом
- При добавлении вкладки в главное окно — дублирование кода недопустимо
- Mixin позволяет добавить `if isinstance(self, QDialog): btn_cancel...` для опциональных элементов

## 10. Почему числовой индекс по оси X в графике (не datetime)

**Контекст:** `chart_window.py`, ось X графика свечей.

**Решение:** `xs = np.arange(n)` вместо timestamp'ов по оси X.

**Причины:**
- Datetime-ось pyqtgraph создаёт пустые зоны для ночей, выходных, клирингов
- Торговые данные имеют нерегулярные gaps (18:50–19:00 клиринг, 23:50–00:00 ночной)
- Числовой индекс = плотный график без разрывов, как в торговых терминалах
- `DateAxisItem` кастомно переводит индекс → дату/время для отображения на оси
